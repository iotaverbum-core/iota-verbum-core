# Demo App Implementation Plan

## Objective

Build and ship a production-quality, casefile-first demo application that replaces the outdated Railway demo and faithfully surfaces the deterministic implemented pipeline:

`EvidencePack -> Claim Proposer -> Claim Graph -> Graph Reasoning -> World Model -> Narrative Renderer -> Ledger + Attestation -> Replay Verification`

## Current State Audit Summary

- Existing backend: FastAPI app in `src/iota_verbum_api/app.py`.
- Existing demo surface: minimal HTML form at `/v1/demo` (outdated).
- Existing deterministic integration: `/v1/casefile/generate` and `/v1/casefile/verify` wrap `proposal.cli_demo.run_demo` and `core.determinism.replay.verify_run`.
- Existing deployment: Docker-based (`Dockerfile`), no dedicated Casefile Studio frontend, no Railway app runbook for this new experience.

## Target Architecture

1. Backend remains authoritative:
   - FastAPI service wraps deterministic core.
   - Canonical artifact generation is backend-only via existing deterministic modules.
   - Replay verification uses `core.determinism.replay.verify_run` unchanged.

2. Frontend:
   - Static single-page UI served by FastAPI.
   - Reads only backend-provided authoritative data.
   - No client-side claims/world/casefile generation.

3. Persistence:
   - Deterministic run outputs stay in `outputs/demo/<run_id>/...` generated by core.
   - API will expose read-only views over authoritative artifacts.

## Implementation Phases

### Phase 1: Backend Casefile Studio API

Create focused endpoints for demo use:

- `GET /api/health`
- `GET /api/fixtures`
- `POST /api/runs/sample`
- `POST /api/runs/upload`
- `GET /api/runs/{run_id}/summary`
- `GET /api/runs/{run_id}/timeline`
- `GET /api/runs/{run_id}/contradictions`
- `GET /api/runs/{run_id}/unknowns`
- `GET /api/runs/{run_id}/receipts`
- `GET /api/runs/{run_id}/artifacts`
- `GET /api/runs/{run_id}/artifacts/{name}`
- `POST /api/runs/{run_id}/replay-verify`

Design rules:

- Reuse `run_demo(..., world=True)` as authoritative generation path.
- Extract run slices from `casefile.json` and `sealed_output.json`.
- Return deterministically ordered collections.
- Add deterministic run progress state reporting tied to real steps.

### Phase 2: Curated Deterministic Fixtures

Add committed fixture dossiers (at least 3):

- timeline-heavy
- contradiction-heavy
- policy/compliance mismatch

Fixture requirements:

- plain text or markdown evidence only
- deterministic, high-signal outputs
- documented expected behavior and trust story

### Phase 3: Casefile Studio Frontend

Build polished static app with:

- Landing page (value proposition + trust story + CTAs)
- Sample case gallery
- Run flow with backend progress steps
- Case workspace panels:
  - verified timeline
  - contradictions
  - unknowns/open questions
  - findings/narratives
  - receipts/evidence trace
  - artifacts/downloads
  - integrity panel
- Replay lab
- Plain-English explainer

Design goals:

- premium, minimal, calm, sharp
- explicit proof-chain visibility
- responsive mobile/desktop layout
- no chatbot framing, no fake streaming

### Phase 4: Determinism and Integration Tests

Add tests for:

- fixture listing deterministic order
- run summary slices deterministic order
- repeat-run consistency for curated fixtures
- replay verification endpoint behavior
- end-to-end API smoke for primary demo flow

Do not alter canonical core outputs; test wrapper behavior only.

### Phase 5: Deployment Readiness

Update deployment assets/docs:

- static asset serving in FastAPI
- startup command compatibility with Railway
- healthcheck endpoint for platform checks
- docs:
  - `docs/DEMO_APP_ARCHITECTURE.md`
  - `docs/DEMO_FIXTURES.md`
  - `docs/DEPLOY_DEMO_APP.md`

## Determinism Guardrails

- No random IDs in authoritative path.
- No wall-clock timestamps introduced into canonical artifacts.
- Canonical JSON and stable ordering preserved.
- Replay verification remains source of truth for integrity.
- App never mutates generated ledger artifacts.

## Risks and Mitigations

1. Existing API auth middleware conflicts with public demo UX.
   - Mitigate by making Casefile Studio routes explicit and controlled in middleware public path handling.

2. Upload flow may introduce unstable file ordering.
   - Mitigate with deterministic temp staging and sorted file handling before `build_evidence_pack`.

3. UI may drift from authoritative data.
   - Mitigate by rendering directly from `casefile` + `sealed_output` slices only.

4. Environment permission issues in test runner temp directories.
   - Mitigate by running required gates and targeted tests in repo-local temp settings where possible; document constraints.
